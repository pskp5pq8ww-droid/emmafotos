Project Overview

This is the complete web portal for Emmanuel Rojas, a wedding photographer based in Brisbane, Australia. It serves two distinct audiences from the same Next.js application:

AudienceURLPurpose
Public visitorsphotographeraustralia.com/Marketing site — about, portfolio, services, contact
Clients/gallery/[slug]Private photo gallery access with PIN login
Admin (Emmanuel)/adminManage clients, galleries, upload photos, settings

What the app does

  • Serves a premium public photography website (English, Australian market)
  • Allows Emmanuel to create private client galleries from the admin dashboard
  • Clients receive a unique URL + PIN to access their private photo gallery
  • Clients can mark favourites, open a full-screen lightbox, and download all originals as a ZIP
  • Photos are processed with Sharp (auto-rotate, 900 px thumbnail generation)
  • All data is stored in a single db.json file + filesystem uploads (no external DB)
  • Deployed on Hostinger as a Node.js standalone app behind Phusion Passenger
Live domain The site is live at photographeraustralia.com. The GitHub remote is pskp5pq8ww-droid/emmafotos.

DB helper functions

FunctionUsage
readDB()Returns full Database object. Ensures data/ dir exists. Creates empty db.json if missing.
writeDB(db)Atomically writes via temp file + rename. Uses a write queue to prevent race conditions.
updateDB(fn)Read → apply fn → write. The standard way to mutate data.
getDataDir()Returns resolved DATA_DIR path.
getUploadsDir()Returns resolved UPLOAD_DIR path.
Adding new entities to the DB To add a new entity (e.g., reviews): (1) Add the TypeScript type in types.ts, (2) add it to the Database type, (3) add it to emptyDatabase with an empty array, (4) add normalization in normalizeDatabase(). Never skip the normalization step or old db.json files without the new field will crash.

Storage on Hostinger CRITICAL

⚠ This section is the most important in the document Getting the storage paths wrong will break all client galleries and lose uploaded photos.

The problem Hostinger creates

Hostinger's Node.js deployment works like this: every time you deploy (push to GitHub + trigger redeploy), Hostinger wipes and replaces the entire nodejs/ folder. This means any files written inside nodejs/ at runtime (db.json, uploaded photos) are deleted on every deploy.

The solution — persistent directories outside nodejs/

All persistent data lives outside the nodejs/ folder, in the home directory which persists across deploys:

Data typeHostinger pathEnv var
Database (db.json)/home/u613502604/data/DATA_DIR=/home/u613502604/data
Uploaded photos/home/u613502604/storage/UPLOAD_DIR=/home/u613502604/storage

Where these env vars are set

They are set in Hostinger hPanel → Node.js → Environment Variables. They persist across redeploys. They must NOT be changed.

⚠ Never change DATA_DIR or UPLOAD_DIR If these values change, the app will create a new empty database and all client galleries will appear broken (photos not found). The old data is still on disk but the app won't know where to look.

Upload directory structure

/home/u613502604/storage/
└── clients/
    └── {client-uuid}/
        └── {gallery-slug}/
            ├── photo-1234567890.jpg        ← original full-res
            ├── another-photo-...jpg
            └── thumbs/
                ├── photo-1234567890.jpg    ← 900px JPEG thumbnail
                └── another-photo-...jpg

How photos are served

The route app/api/files/[...path]/route.ts reads from UPLOAD_DIR/{path} and streams the file. The client gallery requests /api/files/clients/{id}/{slug}/thumbs/photo.jpg. This is why UPLOAD_DIR must be correct.

Bootstrap injection (postbuild)

scripts/postbuild.mjs runs after every build and injects this code at the top of .next/standalone/server.js:

const { mkdirSync } = require("fs");
const _path = require("path");
const _dataDir = process.env.DATA_DIR
  ? _path.resolve(process.env.DATA_DIR)
  : _path.join(__dirname, "data");
const _uploadsDir = process.env.UPLOAD_DIR
  ? _path.resolve(process.env.UPLOAD_DIR)
  : _path.join(__dirname, "uploads");
try { mkdirSync(_dataDir,    { recursive: true }); } catch(_){}
try { mkdirSync(_uploadsDir, { recursive: true }); } catch(_){}

This ensures the directories are created on cold start, even if they don't exist yet.

Deployment flow (full)

  1. Developer runs npm run build locally → .next/standalone/ is created
  2. postbuild.mjs runs: copies .next/static + public/ into standalone, injects bootstrap
  3. Developer pushes to GitHub: git push origin main
  4. In Hostinger hPanel → Node.js → Redeploy (or auto-deploy if configured)
  5. Hostinger pulls from GitHub, replaces nodejs/, starts Passenger
  6. On first request, server.js bootstrap ensures /home/u613502604/data/ and /home/u613502604/storage/ exist
  7. App reads db.json from DATA_DIR — data survives

Authentication System

Admin authentication

File: src/lib/admin-auth/

  • Username: emma (hardcoded in config.ts)
  • Default PIN hash is in config.ts (bcrypt, cost 12)
  • Custom PIN can be set from Settings page → stored in db.settings.adminPinHash
  • Session: HMAC-signed cookie (ers_admin_session), 7-day TTL, httpOnly, sameSite: lax
  • Guard: hasAdminSession() — called at top of AdminLayout

Gallery (client) authentication

File: src/lib/gallery-auth/session.ts

  • Client enters PIN at /gallery/[slug]
  • POST to /api/gallery/[slug]/auth → verifies PIN against bcrypt hash in db.json
  • On success: creates signed cookie ers_gallery_{slug}, 30-day TTL, path scoped to /gallery/{slug}
  • Guard: hasGallerySession(slug, clientId) — called in gallery view page
  • Token format: {clientId}.{expires}.{sha256_signature}
Security note Gallery sessions are scoped by cookie path to /gallery/{slug}, so a session for one gallery cannot access another. Admin sessions are scoped to / (whole site) but the guard is only called in admin routes.

Gallery System

Client flow

  1. Client receives URL: photographeraustralia.com/gallery/natalia-renato
  2. Arrives at PIN login page (app/gallery/[slug]/page.tsx)
  3. Enters 4–6 digit PIN → POSTs to /api/gallery/[slug]/auth

PhotoGrid (masonry)

File: src/components/gallery/PhotoGrid.tsx (client component)

  • Uses CSS columns property — natural aspect ratios, no fixed heights
  • Uses <img> tags (not next/image) to avoid intrinsic size constraints
  • Images fade in on load with onLoad → adds .loaded class
  • Lightbox with backdrop blur on click
  • Favourite toggle: calls /api/gallery/[slug]/favorite
  • Responsive: 4 cols → 3 cols (1100px) → 2 cols (700px) → 2 cols tight (420px)

Image serving

/api/files/[...path] reads from UPLOAD_DIR/{path} and streams the file. Thumbnails are served by default (900px JPEG). The original is also accessible via the full path without thumbs/.
